Introduction

CJD is an embedded documentation technology for Clojure. As such, it's an alternative or supplement to docstrings.

If you're familiar with Javadoc, CJD will look tolerably familiar from a high enough altitude. Basically, the idea behind CJD (or Javadoc, for that matter) is to embed documentation for a program's internals within the source code, in the form of structured comments. These comments are always in the immediate proximity of the artifact being documented, the better to be created, read, and/or updated when artifact is being worked on.

During the normal compilation process, the compiler discards these documentation comments just as it would any other comment, and so they don't affect generated code at all.

As a post-compilation phase in a build process, a CJD function or program makes a pass through Clojure source code, extracts the documentation comments, and from them generates a tree of HTML documents. Interested parties can then view these documents using conventional web browsers.

With respect to documentation comments themselves, the general idea is to provide a simple-minded markup syntax that not only makes styling and formatting possible, but makes salient elements (parameters, return values, methods, etc.) readily identifiable—both to documentation extraction programs and to programmers trying to read the documentation. Consistent with the notion that documentation is being generated for browser-based viewing,  markup includes the ability to easily generate links to points both within and outside of the generated documentation. 

Here's an example CJD-comment, captured live in the wild from the CJD source:

 

#_ (* Processes a sequence of CJD comment-content forms

      into a sequence of flow-forms, which will in turn

      contain phrasing and text content.

      @arg context The context map.

      @arg contents A sequence of input forms to be parsed.

      @(returns

         A tuple of the form @(form [flows remaining]), where\:

         @arg flows A sequence of flow nodes describing

         the parsed content.

         @arg remaining Any unparsed forms in the

              input sequence.)

      )

(defn flow-run [context contents] . . . )

 

You can see a formatted version of this comment here (or here, if you prefer a light background).

A few quick things to observe about this:

·         Following in the Javadoc tradition, CJD uses a specially-formatted comment form to represent documentation comments.

·         CJD associates a documentation comment with whatever it is that immediately follows the comment.

·         Once inside the CJD comment form, it is possible to write something that pretty closely resembles ordinary text—though there are situations where escapes are required.

·         Markup-related utterances are introduced by an @ character.

·         One more subtle feature: Clojure structures are recursively defined… and so are CJD's descriptions of Clojure's structures.

CJD uses the term artifact to denote "a thing that can be documented with a CJD comment". A first-approximation list of out-of-the-box artifacts includes pretty much everything in clojure.core that can be created with a def-something special form, function, or macro. So, artifacts defined with def, defonce, defn, defmacro, defmulti, defprotocol, defrecord, deftype are all natively supported. Other kinds of artifacts include namespaces, and subartifacts representing internal structures of other artifacts, such as a protocol's methods.

But the set of artifacts doesn't—indeed, can't—end there. Clojure, like most Lisp derivatives, is extensible; therefore, it follows that for CJD to be useful as a means of documenting Clojure programs, it must be extensible as well. To this end, CJD does provide facilities allowing programmers to add their own program-specific artifacts. You'll find instructions for writing such extensions in the documented below.

As an article of faith, CJD allows you (in fact, requires you) to exercise your judgment when doing documentation. If some artifacts are documented and others aren’t, that's OK. If you choose, say, to not document a particular function's arguments or return value, that's cool, too.  Regardless, CJD will be grateful for whatever scraps of documentation you throw its way.

For additional commentary on what CJD is and why it is what it is, check out the FAQ.

Getting CJD

CJD is available from Clojars.

By far the simplest method for acquiring CJD and all of its dependent parts and pieces is to use Leiningen—even if you use other methods to build your project or actually run CJD. Add the current CJD descriptor to your project's project.clj file.

For Leiningen 2.x, add it to the :plugins list, e.g.

     :plugins [[cjd "0.1.0-SNAPSHOT"]]

For Leiningen 1.x, this goes in the :dev-dependencies list of development dependencies, like:

     :dev-dependencies [[cjd "0.1.0-SNAPSHOT"]]

Save the project.clj file, then do a lein deps, and Leiningen will automagically download CJD and all of its required libraries and put them in your project's lib/dev directory.

If you can't, won’t, or don't do Leiningen, you'll need to acquire CJD, Extensomatic, Hiccup, and Apache Commons Lang.

Needless to say, CJD also requires some version of Clojure. Currently, CJD works with Clojure 1.2.1 and 1.3.0. CJD does contain version-dependent code, and will likely violently object if you try to use it with other Clojure versions.

Running CJD

Once you've obtained CJD and have a Clojure file with CJD comments, you're ready to use CJD to generate HTML documents.

Before you start, one thing: make sure that your Clojure source files compile cleanly! If Clojure can't load your files, then CJD won't stand a chance either. CJD doesn't care if your files don't run correctly; but it will in general need to load them, so be sure that that they're at least syntactically correct.

When running CJD, generally speaking you'll need to use the same classpath that you'd use for running your program, but with a few additions, notably the jar files for CJD and its dependent libraries.

CJD gives you three main choices for running the document generator: from a REPL or other Clojure code, from the command line, or using Leiningen. All three of these choices are similar in terms of arguments and options, though the details vary.

In all cases, you'll need to specify:

·         One or more Clojure source files and/or directories, which CJD will search for files;

·         A destination directory, which is where CJD will deposit the output HTML documents.

·         Zero or more options.

From the REPL

You can run CJD from a REPL or other Clojure code.

The front-end function cjd-generator is the main point of entry, and provides a comprehensive set of options for controlling the generation process. This function takes a map of options; if you prefer a defnk-style interface, cjd-gen is just what you need.

A REPL session for the CJD documentation itself looks something like:

user=> (use 'cjd.exome)

nil

user=> (cjd-gen ["src/clojure/cjd" "src/clojure/Leiningen"]  

                "doc/dark" 

                :theme :dark

                :title "CJD" :overview "project.clj"

                :exclude [#"src/clojure/cjd/vox.*"]

                :requires 'cjd.cjd.artifact-gen 

                :v #{ :n :f }

                :footer 'cjd.cjd.its-mine/gen-footer 

                )

cjd: processing src\clojure\cjd\artifact_base.clj

  . . .

cjd: processing src\clojure\Leiningen\cjd.clj

cjd: gen-namespace --- #<Namespace cjd.artifact-base>

  . . .

cjd: gen-namespace --- #<Namespace Leiningen.cjd>

user=>

…with a lot of the output elided.

From the command line

You can invoke the CJD processor from the command line, using a command along the lines of

  java –cp <classpath> cjd.main [option...] dest-dir source...

The classpath must include the classes/sources for which CJD. A description of arguments and options can be found here. 

From Leiningen

Set up your project.clj file as described above, and do a lein deps. The downloaded CJD jar file contains the Leiningen plugin; this is installed automatically by virtue of having done the lein deps. You can verify this by running lein with no options; you should see cjd listed as a task.

You configure CJD by using three keys within a defproject within your project.clj file.

·         :cjd-source-path specifies the source files CJD is to process. This can be either a single string or a collection of strings, where each string contains the name of a source file or directory. If :cjd-source-path isn't specified, the plugin uses the value associated with the project map's :source-path, if it's set; and if that's not set, the plugin uses src.

·         :cjd-dest-path specifies the destination directory for CJD output files. If it's not set, the plugin uses doc as the default.

·         :cjd-opts is a map that specifies all other options for CJD.

The plugin uses these three values as parameters to the cjd‑generator function. For details on how they're processed, and on the available options, check out the cjd‑generator documentation.

Once you've got your project.clj file set up, just run lein cjd, and this will fire off CJD processing.

Commenting for CJD

What can be documented

CJD uses the term "artifact" to denote "a thing that can be documented with a CJD comment". A first-approximation list of out-of-the-box artifacts includes pretty much everything in clojure.core that can be created with a def-something special form, function, or macro. So, artifacts defined with def, defonce, defn, defmacro, defmulti, defprotocol, defrecord, deftype are all natively supported.

Some kinds of artifacts have substantial internal structures—in effect, subartifacts—that are also worthy of documentation. Protocols, for example, have methods, and records and types have method implementations. In cases such as these, CJD supports comments not only for the artifact as a whole but for the subartifact as well.

Namespaces are a kind of artifact, too, and can have their own CJD comments. Finally, a project isn't an artifact per se, but can usefully have an overview comment. We note in this regard that a Leiningen  project.clj file is a dandy place to store such a thing.

Of course, clojure.core is merely the stepping-off point for all manner of other potentially documentation-worthy developments, so CJD provides well-defined ways for extending the set of supported artifacts. The effort required to produce such extensions can be near-trivial if your artifact doesn't have documentation-worthy substructure (or can wait to be documented later).

 

The basics

In languages like Java or C, the syntax includes the notion of a "block comment", with a form like /* ..comment text.. */, where what comes between the /* and */ is of more or less of arbitrary content and length. Javadoc leverages this by recognizing comments of the form /** ..javadoc comment text.. */.  Note the leading **—that's the tip-off that it’s a Javadoc comment. Javadoc then processes the comments' contents as Javadoc source material.

Clojure doesn't have block comments as such, but it's got something even better. In Clojure, an entire form can be commented out by inserting the two-character reader macro #_ immediately before the start  of the form. CJD riffs off of Javadoc a bit, and defines a CJD comment as a commented-out list which has a * as its first element. So, if a form of the form

appears in a program, it's a candidate CJD comment.

 

 

Most programming languages come equipped with at least two comment syntax forms, one for single-line comments, and one for block comments. Clojure is pretty much normal in this respect: it uses a ; to introduce single-line comments, and the reader dispatch macro #_ ... syntax for commenting-out entire forms. It's important to note that #_ doesn't exactly give rise to a traditional block-comment structure: it operates on a Clojure form, and the content of that form must be well-structured in the sense that it must meet the requirements imposed by the Clojure reader—that parenthesis and brackets and quotes must all balance, for example. Thus, content of a commented-out form is not quite as arbitrary as it is in, say, a C++ or Java block comment.

CJD makes use of a specialized comment form for representing CJD comments. Riffing a little off of Javadoc, a CJD comment  has the form:

     #_ (* comment-content )

The * must be the first symbol inside the comment form; if it's anything else, the CJD processor assumes that it really is a comment, and ignores it.

Here's a simple CJD comment.

 

#_(* This is a CJD-comment for the var "foo", which

     immediately follows the comment.)

(def foo "bar")

 

There are several things to note about this example. First, CJD associates documentation comments with whatever artifact immediately follows the comment. Next, once inside the CJD comment form, it is possible to write something that pretty closely resembles ordinary text (though there are situations where escapes are required). Finally, this does in fact represent an entirely legitimate CJD comment: while CJD does provide a reasonably complete set of mechanisms for markup, really simple comments are really simple to do.

Here's a little more complicated example.

 

#_(* This is a CJD-comment that describes the function

      foo-fn. It selectively selects one of its arguments.

 

      @p And, it goes into more detail in a second paragraph. The

      blank lines inserted here serve only to make the source more

      readable.

 

      @arg fee This is a description of the first argument.

      @arg fie? This is a description of the second argument.

      @arg foes Describes the remaining arguments.

      @returns Some value if the moon is full, or nil if not.

      )

(defn foo-fn [fee fie? & foe] . . .)     

 

Here we see the beginnings of CJD's markup system, where various aspects of the CJD comment are introduced by tags that follow an @ symbol, such as in the @p or @arg or @returns above, which delimit a paragraph, the description of an argument, and the description of a function's return value respectively..

A few more things to note:

·         The comment above still starts with just plain old text, with no tag in sight. If there's no tag present at the beginning of a comment, CJD views it as the implicit start of a paragraph. This can occur in other situations, too: the @arg tag, for example, looks for a parameter—the argument name—immediately following the tag; then, if there's no tag to specify otherwise, it takes any following content to implicitly be a paragraph. And likewise for @returns, but with no parameters.

·         The example above is formatted in a way to make it fairly easy to read in source form. While this is surely good practice, there's no requirement to do this. CJD uses a lightly hacked version of Clojure reader to read comment content, and consequently has no visibility of any form of white-space content. Thus, like most markup mechanisms, formatting of the comment source has no particular relevance to the formatting of the output.

A final observation. CJD does not currently spend any appreciable effort verifying that a CJD-comment in fact matches what's being documented. Consequently, once inside of the comment, you're able to commit pretty much any mayhem you desire, with no complaints. For example, it's possible—though obviously not very sensible—to place an @arg tag in a comment for a function that doesn't have any arguments, and CJD will not complain.

Occasionally, this lack of checking can be useful. Suppose you've got a state of affairs where you've got a particular form of function that shows up as a parameter in a lot of functions. Obviously, you can document that parameter-function's arguments and return in each of those functions, but that gets old pretty quickly. A better alternative? Write a generic description once, and then use links to refer to it.

CJD markup concepts

To explain CJD's system of markup, there are two principle concepts that you need to be aware of.

First,  CJD specifies markup in terms of its own system of elements. In its full glory of its most general syntax, an element is introduced by the @ symbol, followed by a list, the first element of which is called the tag:

     @(tag argument... content)

Items in the list that follow the tag may be arguments of the element, or "content", which is a sequence of forms of arbitrary length. What's required and/or acceptable for both arguments and content varies considerably depending on the specific tag. Some tags have no arguments, others no content, and a few have neither.

In some cases, this syntax is necessary. In others, CJD is able to delimit an element's scope by other means, without the need for full list syntax. In these cases, it's sufficient to write the element without the parentheses, i.e., as @ followed by the tag, followed by any content.

So, for example, we could write out a couple paragraphs as

     @(p This is the first paragraph.)

     @(p This is another paragraph.)

Or, we can optimize things a bit, and get the same results by specifying:

     @p This is the first paragraph.

     @p This is another paragraph.

…where the second @p serves to delimit the first one.

Or mix-and-match, as needed:

     @p This is the first paragraph.

     @(p This is another paragraph.)

We call the full list version a closed-form element, and the unparenthesized version an open-form element.

This begs the rather complicated question of just when it is that you can get away with using the open form, vs. when you must use the closed form. We’ll talk about this some later. For now, the essential takeaway is that elements do come in two related-but-different syntactic forms—so don't be surprised when you see, say, @(arg ...) in one instance and @arg ... in the next; they're both talking about the same thing.

 

The second major concept involves how CJD organizes the content of a comment.  CJD works on basis of a simple-minded model that vaguely resembles that of HTML. There are two content domains: on one hand, phrasing content consists of text—words and punctuation and links and the like, along with markup for that text, like styling (like bold or italic). When processed, phrasing content conceptually gives rise to a long, one-dimensional string, without any particular indication of how that string is placed on a page or organized into higher-level structures.

Flow content, on the other hand, essentially consists of containers that occupy area on a page. Specific containers might contain phrasing context—a paragraph, for example—or might contain other containers. 

Specific elements give rise either to phrasing content or to flow content. A phrasing element is all about text, and generally creates or styles text. For example, the @i and @b elements specify that some specified phrasing content is to be rendered as italic and bold, respectively, @link and @linkto elements insert links into the phrase stream, and the @name element inserts the formatted name of the source code artifact to which a CJD comment applies.

Flow elements organize comment text, and provide the structural metadata that allows a comment to be meaningfully formatted. For example, the @p denotes a paragraph (which pretty clearly defines a particular grammatical structure that's salient to the formatting process), and the @arg and @returns elements describe flows that provide descriptions for arguments and return values respectively. 

Flow elements generally have a recursive definition, which is to say that a flow element can contain other flow elements. The one exception to this is the @p element, which makes a slightly different assumption: if its content starts with a flow element, then it acts as a container for a sequence of flow elements. However, if its content starts with text or a phrasing element, then it contains phrasing content—and thus it's the main bridge between the phrasing and flow domains.

CJD can often infer the start of paragraph element even if @p isn't specified. In general, if the content of a flow element doesn't start with another flow element, CJD simply assumes @p is implicitly there. For example, suppose we want to document an argument named thud. For this we'd use an @arg element, which we might write as:

     @(arg thud @(p This is the description of thud.))

Or, as an open-form equivalent:

     @(arg thud @p This is the description of thud.)

Or, we can let CJD infer the paragraph:

     @(arg thud This is the description of thud.)

And finally, we can use an open-form representation for the @arg element:

     @arg thud This is the description of thud.

In this case, all of these variations would work equally well.

 

So, where do we need to use closed-form elements? Where would an open-form element not work? Fundamentally, it's an issue of associativity.

Here's an example. Suppose that the description of thud needed two paragraphs. In full closed form representation, we'd have:

     @(arg thud

         @(p This is the first paragraph.)

         @(p And here is the second paragraph!))

Here we can legitimately reduce the @p elements to open form:

     @(arg thud

         @p This is the first paragraph.

         @p And here is the second paragraph!)

Now, if we convert the @arg element to open form, we'd get

     @arg thud

         @p This is the first paragraph.

         @p And here is the second paragraph!

Although this is perfectly acceptable, syntactically speaking, CJD will interpret it as

     @(arg thud

         @p This is the first paragraph.)

     @(p And here is the second paragraph!)

Depending on the circumstances, this might give rise to some very different (and possibly misleading) formatting.

In general, you'll probably want to limit the use of open-form tags to simple situations. If you're feeling expansive or complicated, you will probably need to use closed-form tags.

Comment content, and dealing with the reader(s)

When it comes to dealing with comment text, CJD by in large attempts to generate output that matches the comment text, so that, modulo influences of the various markup elements, what you see in the source is pretty much what you're going to get on the output. Consequently, CJD renders symbols, names, lists, vectors, strings, and keywords in the source text mostly verbatim on output. So, for example, a parenthesized remark in comment text is read in as a Clojure list—but CJD knows to make it appear as a parenthesized remark on output.

Ultimately, this WISIWYG-ish behavior is constrained from several directions. For one, one really basic requirement of the CJD system is that CJD comments must be 100.00% compatible with the stock-standard, unmodified, straight-out-of-the box Clojure reader. Because the reader processes a CJD comment exactly as it would any other form, there are a few constrains on what can go into a CJD comment—all of which provide scope for unpleasant gotchas.

And the fun doesn't stop there. CJD internally uses a lightly-hacked version of the Clojure reader to process CJD-comments (though we can see our way to rectifying this in the future). It differs from the Clojure reader, for example, in recognizing CJD-comments as data-containing forms, not comments, and not discarding commas. Modulo these differences, the CJD reader converts comment input into lists, vectors, maps, symbols, names, strings, sets, numbers, etc., in exactly the same way that the Clojure reader does.

Thus, if you're trying to enter plain old text, reader behavior provides a plethora of ways to get into trouble. Here are a few common examples:

·         Semicolons are comment characters. To escape a semicolon, use \;.

·         Colons can only live at the start of a word—the reader expects the colon to be the start of a keyword, after all. To escape a colon, use \:.

·         Slashes are special. A standalone slash (as for the divide operator) works OK (it's a divide-by operator, right?), but it has to be cleanly delimited from other text. If you want to write something like "and/or", either use the literal insertion syntax (see below), or, if your sense of typographic fitness can tolerate it, use spaces, as in "and / or".

·         Punctuation is often tokenized as part of the preceding word. For ordinary text, this isn't a problem, but it can cause problems when using open-form tags. For example, it's easy to type @name. at the end of a sentence. The reader is OK with this, but CJD will flag it as an unrecognized tag. Either @name . or @(name). or @name\. will work fine, though.

·         Numbers followed by punctuation are a variation on the above: 1, for example, currently causes the reader to complain. Just insert a space, as in 1 , and CJD will discard it.

If all else fails, CJD provides a literal insertion syntax: a string immediately preceded by a tilde causes the contents of the string to be inserted verbatim into the output text. Thus, ~"#@%$!" will indeed cause #@%$! to appear in the output (the string is HTML-escaped, however—CJD doesn't currently allow insertion of literal HTML code).

Another consequence of using a reader for input is that CJD has no visibility of any formatting that may have been applied to the comment source (strings within CJD-comments follow normal Clojure conventions, so they're a very limited exception). Aside from this, no source formatting is preserved (but that's what markup is for, right?).

This implies CJD has an interesting job taking the stream of stuff from the reader and putting it together into a form that looks like it should. This involves making some decisions about how the various bits and pieces go together: for example, if CJD sees a sequence of words, like sequence of words in the input, it knows to insert space between the words. Similarly, if it sees a comma or period, it knows to append that to the output text.

Sometimes, however, you may need to change the default behavior. To this end, CJD interprets several keywords found in comment text as directives to ensure that the opposite behavior occurs:

Table 3 -- In-text keywords interpreted by CJD

Keyword      

Action

:nb

:nobreak

Causes CJD to suppress inserting space between two words, symbols, or elements that it would ordinarily separate.

:br

:break

Causes CJD to insert space between two words, symbols, or elements in situations where it would normally run them together.

So, for example, suppose we have input text of the form

@(i Stu) @(u pen) @(i dous!)

By default, CJD will insert spaces between each element, giving a result like Stu pen dous!, whereas what we'd really like is Stupendous! You can get this by using

@(i Stu) :nb @(u pen) :nb @(i dous!)

using the :nb keywords to suppress the normal spacing behavior.

 

Writing CJD comments

Comment structure

There's a fair amount of latitude in what goes into an acceptable CJD comment—CJD is fairly tolerant of what it's given—but there is a nominal structure for comments that's likely to lead to the most predictable results.  

Here's a first example of a complete comment.

#_ (* Returns the number of items in a collection.

 

      @p @name works regardless of whether @(arg coll) is a set,

      vector, list, map, or string.

      

      @arg coll The collection to count.

      @returns The number of elements in @(arg coll). Returns 0 if

               @(arg coll) is nil.

      @since Clojure ~"1.0".

      )

(defn count [coll] . . .)

We'll look at this example in more detail shortly, but for the moment, let's just look at the overall structure.  

·         The first paragraph is (somewhat rudely) designated as the blurb, and is notionally a short summarization of what the artifact is or does. CJD uses the blurb as an abstract of the full documentation, using it, for example, in summaries of the content of namespaces. Note that we didn't bother inserting the @p one would typically expect to see starting a paragraph—it's an example of CJD implicitly determining that a new paragraph has begun.

·         The blurb paragraph is followed by zero or more description paragraphs. These provide detailed information about the artifact. Here's where you can wax eloquent (or poetic, or whatever else suits your fancy) on what your artifact is or does, or why or how it does it, or whatever other else you think is important about it. Here there is a need to start each paragraph with @p.

·         After the description, datum-descriptions provide information about data elements relevant to the artifact—arguments, options, return values, and the like. These are items for which it's desirable to have some documentation, but they're strictly local in scope, and don't warrant a CJD-comment of their own (unlike, say, methods of a protocol, which give rise to artifacts that are global in scope and do warrant full documentation). For example, a CJD-comment for a typical function would include a description for each parameter named in its argument list, plus a description of the function's return value. There are a number of elements that introduce data descriptions, including @arg, @opt, @option, @popt, @nopt, and @field.

·         Finally, info-descriptions provide supplemental information about the artifact—authors, initial release, references to other sources, deprecation status, etc., corresponding to @author, @since, @see, and @deprecated elements.

We might summarize this structure as follows:

#_(* blurb-paragraph

 

      description-paragraphs...

 

      datum-descriptions...

 

      info-descriptions...

      )

It's probably possible to depart from the ordering above, and may even make sense to do now and again... but: (1) something will happen if the order is changed… and it might even be what you expect! and (2) bear in mind that there is some value in adhering to a consistent ordering as it makes information that much easier for a reader to find.

The blurb paragraph is pretty much the only thing that really needs to be present in every CJD comment. Everything else is optional, to be used as circumstances and discretion require.

Documenting a function, and the joys of recursive documentation

A very simple function

Functions are the sine qua non of Clojure, so we'll use functions as the basis for introducing some of CJD's important capabilities. This notwithstanding, these capabilities apply equally well to any species of parameterized artifact, such as macros, methods, multimethods, and the like.

Here's the previous example with a different set of emphases.

#_ (* Returns the number of items in a collection.

 

      @p @name works regardless of whether @(arg coll) is a set,

      vector, list, map, or string.

      

      @arg coll The collection to count.

      @returns The number of elements in @(arg coll). Returns 0 if

               @(arg coll) is nil.

      @since Clojure ~"1.0".

      )

(defn count [coll] . . .)

We've already seen @arg and @returns as a way of starting the descriptions for the arguments and return values for a function., but there's also a couple of new things here.

·       @name is a shorthand for the name of the artifact being documented. Whenever it occurs, CJD inserts an appropriate rendering of the artifact name into the output.

·       We've seen @arg as a way of introducing argument descriptions, but CJD interprets the special case  @(arg name)— specifically, where name is exactly a single identifier—as an indication that it should insert an appropriate rendering of name into the output.

So, what's an "appropriate rendering"? CJD keeps track of the identifiers used in and around a CJD comment, noting, in particular, the context in which they're defined. This information drives the rendering process. If you look at the output generated for the comment above, you'll note that CJD chooses one color for the name of function, and another color for the parameter coll.

There's a subtle point that needs to be made about what CJD does here. CJD recognizes an identifier used within a CJD comment only when that identifier is documented, i.e., when there is an @arg (or other) datum-description for the identifier. In effect, for CJD's purposes, an identifier's documentation is its "declaration", and that's what determines its "appropriate rendering". In particular, identifiers in a function of method signature are not declarations, from CJD's viewpoint.

One (often glaringly apparent) effect of this is that the names of parameters that aren't documented (or for which the name is misspelled) appear in the normal foreground color in the output. It's a hint from CJD that there's a problem.

More complex data

Obviously, the function above is a rather simple case. Many programming situations demand that not-so-simple structures be passed to or returned from functions. Formal structures, like records, types, or classes, are one way to do this. But Clojure allows data to be informally structured as well, often on an ad-hoc basis.

For example, to return several values from a single function call, all the programmer need do is to compose those values into a vector or list, and return that single object. So, let's consider a function that simultaneously generates the min and max values of a collection, and returns them as a two-element vector:

 (defn minmax [coll] [(apply min coll) (apply max col)])

A simple enough function; but how would we document this? The function only returns a single value… but it's really two values! Here's how to document this in CJD.

#_(* Computes the min and max values of a collection.

      

      @arg coll A collection of numbers.

      @(returns A tuple of the form @(form [minval maxval]), where\:

         @arg minval The minimum value of @(arg coll).

         @arg maxval The maximim value of @(arg coll).)

     )

(defn minmax [coll] [(apply min coll) (apply max col)])

Note the @(form [minval maxval]) element: it's the key here. In effect, this is saying that the single return value of the minmax function is a construct—putatively a vector in this case—that contains two elements. So that we can conveniently talk about them, we use the @form instance to give these elements names, namely minval and maxval. Then, we can document them using the same @arg elements we've already encountered. This essentially the same process we'd use if minval and maxval were parameters of a function—but here it occurs within the scope of the @returns element, because that's where @form instance that declares them lies.

Naturally, this works for parameters, too:    Dark Light

#_ (* Returns the average of the values in a min / max tuple.

     

      @(arg min-max A tuple of the form @(form [minval maxval]), where

            @arg minval A number representing a minimum value.

            @arg maxval A number representing a maximim value.)

      @returns The average of @(arg minval) and @(arg maxval).

      )

(defn average [min-max]

  (let [[minval maxval] min-max]

    (/ (+ minval maxval) 2)))

And this works not only for parameters and return values, but pretty much everywhere.  

And away we recurse…

Clojure's data structures are recursively defined, so it follows that a proper means for documenting those structures must be recursively defined, too. CJD supports this by allowing the structure of any value defined in any context—even ad-hoc definitions created for expository purposes, such as @form elements—to be elaborated as needed.

 

#_ (* Generates a curve through a set of points.

            where

                  where

                  @arg y The y-coordinate.

                  @arg z The z-coordinate.)

                  the following\:

Here we're using several @form elements to describe data inside of data being described by another @form. The nesting of the elements is what keeps all this sane, just like with Clojure code.

Just for variety, we've chosen to express the alternatives for the color value as a bulleted list. This uses the time-honored @ul and @li elements (yes, there is a @ol element for numbered lists, too). The essential restriction is that any @li element must reside within its parent @ul (or @ol) element.

Maps

Functions

Last, but by no means least, we come to functions as data—the notion of functions as "first-class" objects. Functions frequently occur as data in Clojure, in the form of parameters passed to or returned by other functions, or as elements of data structures. The fact that these functions aren't defined as top-level artifacts makes them no less worthy of documentation than their top-level counterparts. After all. a function that requires a function as a parameter often has very specific expectations about the form of the parameter-function—and a programmer coding a call to that function surely needs to know about those expectations.

CJD addresses this by allowing functions-as-data to be visualized and documented in a fashion similar to that used for data. Here's an example, based on a (not-necessarily-accurate) version of the familiar reduce function.

#_ (* Generates and returns a single value by applying a function

      to the elements of a collection.

     

      @p @name internally maintains a value that serves as an "accumulator",

      which is initialized to @(arg val). Then, @name calls the function

      @(arg f) once for each element of @(arg coll), passing to it the

      current accumulator value and the next element of @(arg coll).

      @name then updates the accumulator with the value returned by @(arg f).

      When @(arg f) has been called with all elements 

      in @(arg coll), @name returns the final value of the accumulator.

     

      @(arg f A function of the form @(fun [acc element]), where

            @(arg acc The accumulator value, which is @(arg val) on the

                  first call to @(arg f), or the result of the prior

                  call to @(arg f) on subsequent calls.)

            @(arg element The next unprocessed element of @(arg coll).)

            @returns The updated accumulator value. )

      @arg val An initial value for the accumulator.

      @arg coll A collection of elements.

      @returns The value returned by the final call to @(arg f). If @(arg coll)

      is empty, returns @(arg val).

      )

(defn reduce [f val coll] . . .)      

Documentation for the function f pretty much follows the pattern already established. It's documented as a parameter in the @(arg f ...) element, and structurally described and its parameters named using the @(fun ...) element. Then, @arg and @returns elements describe f's interface. As with other recursive documentation cases, these  @arg and @returns elements must occur within the @arg element that describes f.

Needless to say (but worth saying nonetheless) CJD implements all this in a fully general, fully recursive way. A function described by @(fun [...]) can have parameters or a return value that are functions or complex data objects or maps that are described by the techniques we've already discussed..

Documenting records, protocols, and their friends

 

Linking to other stuff

As you've no doubt noticed, one of the things that makes hypertext so useful is the ability of one document to link to other documents, or even to other locations in the same document. CJD is well and truly on this bandwagon, and provides several link-generating elements to make it as easy as possible to insert links whenever the opportunity arises.

CJD links come in several forms. The most general-purpose of these is the linkto element, which can be abbreviated as merely lt. It has one required argument—a string containing a URI—and can optionally be followed by phrasing content that, if present, forms the user-visible manifestation of the link. So, for example,

    @(linkto "http://www.example.com/whatever" the @(i good) stuff)

generates a link to the indicated site in the output document, with "the good stuff" as the displayed text.

While linkto is most general, it's also the most work, and so is best used for linking to other documents. Experience suggests, however, that within a CJD comment, most useful links are likely to be to the documentation of other artifacts within the scope of the artifact being documented. For example, if you're writing a CJD comment for a function, your comment may well reference other functions, vars, or named entities within the same namespace or in a namespace loaded via use or require, or a class locally created (as by defrecord) or imported. CJD therefore aspires to make adding links to such named entities very easy to do.

And so the link element (and the link tag can be abbreviated simply as l), or, more precisely, @(link linkable). The parameter linkable should be a symbol, and more specifically, needs to be symbol that is the name of an entity that's accessible within the current namespace. The name given in linkable follows all of the normal rules Clojure defines for name resolution. If you've defined the name in the current namespace, or imported it with use, you can specify just the name; otherwise, you'll need to specify the fully-qualified or prefixed name, exactly as you would in your code. So:

 

(declare abc-xyz)

 

#_(* This is documentation for function @name. It lazily

      relies on @(link abc-xyz) to do its work for it.

(defn xyz-abc [] (abc-xyz))

 

#_(* Does something remarkable with @(l xyz-abc).

      )

(defn abc-xyz [] (xyz-abc))

 

Well, don't try running it… but CJD will resolve the links between the comments correctly.

The link element resolves linkable in the context in which it's defined, to end up with a fully qualified name, which can be of a namespace, a name within a namespace, a class, or a member of a class. Then, CJD tries to translate this name into a URL for the named entity's documentation.

If the name is "locally-defined"—i.e., defined in any namespace being processed by the running instance of CJD—then CJD will recognize that fact and insert a URL to a document in the tree being generated. Similarly, if the namespace is well-known (where "well-known" currently means clojure.* and java.*) CJD generates URLs that reference documentation pages on appropriate public sites.

This leaves the case where a name is in a namespace that's neither locally-defined nor well-known. To deal with this, CJD provides the means by which a determined CJD documenter can add link resolution functions. See the documentation below for details

 

Now, here comes a complication that specifically relates to linking to documentation of records, types, and entities that are built on top of them (like CJD's internal artifact objects, for example). Clojure naming turns out be… a little tricky. Specifically, Clojure operates using two related-but-distinct naming conventions: its own, and Java's.

On one hand, when you define, say, a function my-fn in a namespace our-ns, defn creates a var with a "fully qualified" name of the form our‑ns/my‑fn. If you're within our-ns, or have imported it via use, my-fn will happily resolve just as expected. Likewise, if our-ns is loaded by requires, with or without a prefix, my-fn can be referenced as a fully qualified or prefixed name, as appropriate. All this is perfectly normal business-as-usual, and the link  element will resolve any such name without problem; and so @(l my-fn) should produce a link to my-fn's documentation.

On the other hand, if you use, for example, defrecord to create a record with a name Foo in our-ns, Clojure creates a class that's named using the Java conventions, our_ns.Foo (and note that the class name is a munged version of the nominal Clojure name!!). In this case there is no entity named our-ns/Foo, and references to it cause loud complaints from Clojure—and from CJD, if you try to link to it. 

The unpleasant consequence of this is that it's generally not possible to reference the record as just Foo, as in the example. In general, you'll need to specify this as @(l our_ns.Foo), or even better, as  @(il our_ns.Foo), which will reduce the name so that just the text "Foo" appears for the link.

The CJD context object

CJD uses a "context object" ubiquitously in its internal operations. It's defined by the Context record type, and essentially consists of a long list of fields and what amount to setter/getter methods.

The context object contains essentially all of the global state information for a CJD run—everything from option settings to the start time of the CJD run, but parts of it are updated on the fly to describe what's currently being processed. It also contains methods for generating context-sensitive messages, warnings, and errors.

Unless you're really digging into the internals of CJD, you probably won't need to deal much with the context object; but many CJD functions require or receive it as a parameter. This is often done on a pro-forma basis; so the takeaway here is that if you need, say, to write a function that takes a context object as a parameter, and your function just ignores it, then that's OK—it won't bite.

Link resolver functions

One of CJD's goals is to make it easy to link to documentation of program-related entities outside of the immediate namespaces being documented, such as Clojure functions, Java classes, or indeed the contents of just about any of the myriad packages accessible through Clojure. CJD natively knows about a few such bodies of documentation, notably clojure.core and the standard JRE java classes. Thus, if you're documenting a function that uses, say, clojure.core/extends,  you can just put @(l extends) in a CJD-comment and CJD will insert the link to Clojure's extends documentation (as we discuss in detail above).

To make this happen, CJD first resolves the symbol to its fully qualified name, which it does by normal Clojure mechanisms. Then, it needs a way to map the fully qualified name into its documentation's URL. This is where link resolver functions come in: CJD sequentially calls each link resolver function it knows of, passing it the symbol's namespace and name, until one responds with a URL.

Thus, a link resolver function needs to do two things: first, to decide if it recognizes the namespace and/or name it's handed (and beware: the name can be nil if it's just a namespace being named!), and second, if it does recognize them, to generate an appropriate URL. A link resolver function can be pretty much as general or specific as desired: for example, CJD comes equipped with a couple of default name resolver functions, one of which recognizes  clojure.* namespace, and the other, java.*.  

With link resolver function in hand, call CJD's add-resolvers function to add your resolvers. Here's an example, which includes CJD's Clojure link resolver function:

 

(ns my-project.cjd-fns

  (:require [cjd.link-resolver :as res]))

 

(res/add-resolvers

     (fn [ns sym]

      (if (re-matches #"clojure\..*" (name ns))

        (str "http://clojuredocs.org/clojure_core/" (name ns)

             (if sym (str "/" (name sym))))))

 

Then, when running CJD, inform CJD about this namespace using the :requires option, as described above.

CJD calls resolvers in the reverse order that they were added to the resolvers list. Consequently, you can effectively override a resolver by adding another that recognizes the same namespace. Alternatively, you can just chuck out all existing resolvers by calling reset-resolver-fns — presumably before adding any resolvers of your own.

Document headers and footers

CJD can place simple headers and footers the documents it generates. You might like replace or add to these headers or footers, perhaps to add a logo, or a copyright or other fine print (as, for example, in the footer that appears on each page of CJD's documentation).

If you'd like to replace, or add to, the existing headers, write a function that generates the HTML for the header or footer. Here’s an example of a footer function that uses the Hiccup package to add a copyright message; it's derived from the namespace used by CJD for its own documentation.

 

(ns my-project.cjd-fns

  (:require

[cjd.generate :as gen]

[cjd.context :as ctx]

[cjd.custom :as custom]

    [hiccup.core :as hiccup]))

 

(defn gen-footer [context]

  (hiccup/html

    (gen/gen-trailer context)

    [:div {:style "text-align: center; width: 100%;"}

      [:span {:style "font-size: 8pt;"}

          "Copyright 2012 MegaCorp Inc. All rights reserved."]])))

 

(custom/set-footer 'gen-footer)

 

In this case, the gen-footer function adds to, rather than replaces, CJD's default footer by calling the gen-trailer, the default footer generator; but this is entirely optional. The corresponding default header function is gen-leader.

After defining a header or footer function, you've got two options: you can either call cjd.custom/set-footer and/or cjd.custom/set-header, as appropriate. Alternatively, you simply specify the function(s) when you run CJD, by using the :header and :footer options. In either case, you need inform CJD about the namespace containing your header/footer functions using the :requires option, as described above.

Extending CJD's artifact model

CJD maintains a model—in the object model sense—of the entities for which it generates documentation. Its out-of-the-box object model encompasses pretty much all of the entities defined by def-something forms in Clojure, including functions, macros, vars, records, protocols, multimethods, and so on. A few additional objects, notably namespaces, are defined as well.  

CJD collectively labels these entities as "artifacts", and its object model, unsurprisingly enough, is essentially a shallow hierarchy rooted at an abstract artifact object.

Quick start: adding support for a simple artifact

Here's a typical scenario. Suppose you've written a macro called defwhatsit, which you then invoke whenever you feel it's appropriate to create a new whatsit. So, we'd expect to see something like

 

(defmacro defwhatsit [name other-parameters...]

   ...implementation... )

 

Here's an example of what minimal CJD support might look like.

 

(ns my-ns.cjd

  (:use

    [cjd.artifact-base]

    [cjd.generate]

    ))

 

(defartifact Whatsit defwhatsit "What Is It" [][]

             simple-artifact-parser)

 

(mk-category :whatsit "What Is It")

 

(mk-art Whatsit gen-desc :whatsit)

 

Here's what it means:

·         The defartifact call defines a new type of artifact, which ultimately is represented as a Clojure record type with a canned set of fields and methods supplied by defartifact. The parameters here include:

o        Whatsit, a name for the artifact (any valid Clojure name will do).

o        defwhatsit, the name of the function or macro that defines a whatsit. This is the symbol CJD uses to recognize that a whatsit is being defined.

o        "What Is It" is a (very) short string describing of what kind of thing the artifact represents, available for potential use in describing the artifact in the output.

o        The empty vectors [][] would come into play if the Whatsit artifact needed additional content (we'll get to that anon); but here are just placeholders.

o        simple-artifact-parser is a very simple-minded parser that parses each defwhatsit instance to the extent of extracting the name of the whatsit.

·         The mk-category call is optional; it defines a new category of artifact. The primary place this gets used is in producing the "summary" lists found at the start of each namespace. The call here assumes that whatsits are sufficiently different from all other kinds of artifacts so as to warrant creation of a new category, i.e. that it's not like a function or record or anything one would find in core Clojure.

o        :whatsit is a key identifying the category.

o        "What Is It" is the category name (you'd expect to see a What Is It Summary section in each namespace in which it was used).

If you think your artifact fits well enough into an existing category then this call isn't needed. Categories for core Clojure artifacts are defined in the cjd.generate

·         The mk-art call tells the output generator how to handle whatsit artifacts. Parameters include:

o        Whatsit is the artifact record name, as defined in the defartifact call.

o        gen-desc is a simple-minded output generation routine. It assumes that the artifact has no substructure (arguments, options, methods, etc.) of any kind, and so just generates HTML for the CJD comment text with no further ado.

o        :whatsit is the category key defined by mk-category.

…and that's it.

To get CJD to recognize your additions, run CJD with a classpath that includes the my‑ns.cjd namespace, and use the :requires my‑ns.cjd option (or ‑‑requires my‑ns.cjd from the command line) to commend the namespace to CJD's attention.

Defining more complex artifacts

When it comes to artifacts, sometimes "simple" is just not enough. Complicated artifacts often contain substructures or specificational elements that need to be recorded and reflected in their documentation, which gives rise to a need for more complicated structures to record them.

CJD uses a small package called Extensomatic to generate the data structures it uses to represent artifacts. It's build atop Clojure's standard protocol and record mechanisms, and is principally aimed at allowing simple structures to be conveniently and concisely defined. In the end, Extensomatic offers no capabilities not already available in Clojure, but merely provides an arguably more convenient way of specifying simple data structures than defrecord manages to attain.

Here's the quick and dirty description of Extensomatic. It's based on two kinds of artifacts, called extensos and constructos. An extenso is a fragment of a record; it's specified by the defextenso macro. An extenso can define fields (and optionally, initialization values for those fields) and methods and their implementations. An extenso can also include other extensos, inheriting the included extensos' fields and methods. All methods and field initializations are final, i.e., no overrides are allowed, although an extenso can specify an initialization value for any uninitialized field of an extenso it includes. FWIW, extensos vaguely resemble traits or mixins, although the details vary extensively.

Where an extenso defines a fragment of a record, a constructo, specified using the defconstructo macro, defines a real, live, instantiable Clojure record. A constructo can define fields, field initializations, and methods, and can include extensos just as an extenso does. In practice, defconstructo gathers up all of the contents of a constructo's included extensos, adds the constructo's locally specified methods and fields, repackages it as a Clojure record, and passes it off to the tender mercies of defrecord, and voilà! A record is born.
