... on what CJD is and does

What is CJD?

It's an embedded documentation technology for Clojure, an alternative to docstrings. If you're familiar with Javadoc, CJD will look tolerably familiar from a high enough altitude.

Basically, the idea behind CJD (or Javadoc, for that matter) is to embed documentation for a program's components within the source code, in the form of structured comments. These comments are always in the immediate proximity that code that's being documented, the better to be created, read, and/or updated when the documented code is being worked on.

During the normal compilation process, the compiler discards these documentation comments just as it would with any other comment, and so they don't affect generated code at all.

As a post-processing phase in a build process, a CJD function or program makes a pass through source code, extracts the documentation comments, and from them generates a tree of HTML documents. Interested parties can then view these documents using conventional web browsers.

With respect to documentation comments themselves, the general idea is to provide a simple-minded syntax that makes salient elements (parameters, return values, methods, etc.) readily identifiable---both to processing programs and to programmers trying to read the documentation---along with a simple-minded form of document markup.

How would I know a CJD comment if one bit me?

In languages like Java or C, the syntax includes the notion of a "block comment", with a form like /* ..comment text.. */, where what comes between the /* and */ is of more or less of arbitrary content, can span multiple lines, and can run on for as long as the commenter likes. Javadoc leverages this by recognizing comments of the form /** ..javadoc comment text.. */ (note the leading **---that's the cue), and then processing their contents as Javadoc source material.

Clojure doesn't have block comments as such, but it's got something even better! In Clojure, an entire form can be commented out by inserting the two-character reader macro #_ immediately before the start  of the form. So, CJD riffs off of Javadoc a tiny little bit, and defines a CJD comment as a commented-out list which has a * as its first element. So, if a form of the form #_(* ...contents...) appears on your radar, it's a candidate CJD comment.

Here's a genuine certified example of a simple CJD-comment, captured live in the wild:

 

#_ (* Processes a sequence of CJD comment-content forms into a

      sequence of flow-forms, which will in turn contain phrasing

      and text content.

      @arg context The context map.

      @arg contents A sequence of input forms to be parsed.

      @(returns A tuple of the form @(form [flows remaining]),

         where\:

         @arg flows A sequence of flow nodes describing the

         parsed content

         @arg remaining Any unparsed forms in the input sequence.)

      )

(defn flow-run [context contents] . . . )

 

You can see a formatted version of this comment here (or here, if you prefer a light background).

This isn't the place for describing the gory details of what's going on here; but several quick thoughts:

§         The comment comes before the artifact it describes, not in the (yucky) middle, like docstrings. (We stipulate that this is pure undiluted prejudice on our part.)

§         The forms beginning with @ are what CJD uses both for structuring and marking up the comment. Depending on circumstances, these can be either open-form (like the @arg elements) or closed-form (like the @(returns ...)), and may have parameters or define a range of text.

§         Markup notwithstanding, once you're inside the CJD comment form, it's possible to write something that pretty closely resembles ordinary text---though there are situations where escapes are required.

§         Documentation relating to elements of interest to a reader---like arguments, return values, etc.---is fairly easy to locate.

§         Finally, FWIW, the return value described in the comment above is a nice example of a recursive documentation structure, wherein we take a single datum (the return value) and elaborate on its structure.

For the gory details, see the user guide, which was laboriously prepared to belabor all the details.

Can I put any just old thing in a CJD comment?

Close, but not quite. There's been a pretty serious effort to make the source content of CJD comments read more or less like normal text, and generally speaking most of your literary efforts will fit in pretty much as you'd ordinarily type them.

But, bear in mind the following: a CJD comment is a Clojure form. It's a commented-out form, to be sure, but a form nonetheless. During normal program compilation, the Clojure reader will subject this form to all the scrutiny that it applies to any other form. So, if things that make the reader go bang! appear in a CJD comment, then bang! it will go.

Like, paired characters---parens, brackets, braces, quotes, etc.---all have to match up correctly, just like in fully compiled forms. Like, if your comment says "see the following:", the reader will nastily inform you that it really, truly does not like colons that occur at the ends of words. An unescaped semicolon will cause everything to the next end-of-line to be flushed (which gives rise to the occasionally-useful notion of commenting out comments). To deal with these and other situations, CJD supplies several escape mechanisms (see here).

But empirical experience suggests that these little gotchas don't really interfere with one's ability to write or read more or less normal-looking text. And after a while, typing stuff like it does that\; or maybe this\: comes to feel perfectly... uh... normal.

Does CJD insist that I comment everything?

Nope. When it comes to what material you choose to document for an artifact, CJD is entirely undemanding. If you document a function, say, you can choose to document all or any or none of the parameters, or to document the return value---or not.

To make this a little more explicit, CJD currently does not do any enforcement at all. In some cases, this may even be a little too much of a good thing: you can document parameters, for example, that are not actually declared by a function; there's nothing to ensure that comment content is in sync with what's being commented upon. But if you have problems keeping things in sync without a program holding your hand, you're not gonna cut it as a Clojure programmer anyway.

In general, though, CJD assumes that its users are competent programmers capable of exercising good judgment about what needs to be documented and what doesn't. Consequently, CJD will pretty generally gratefully accept whatever scraps of documentation you feed it.

What kinds of things can be documented with CJD?

We’ve taken to using the term "artifact" to denote "something that can be documented with CJD". A first-approximation list of out-of-the-box artifacts includes pretty much everything in clojure.core that can be created with a def-something special form, function, or macro. So, artifacts defined with def, defonce, defn, defmacro, defmulti, defprotocol, defrecord, deftype all work, and probably a few others that don't come to mind at the moment.

Some kinds of artifacts have substantial internal structures---subartifacts---that are sufficiently notable to be worthy of CJD documentation comments above and beyond their containing structures. Protocols, for example, have methods, and records and types have method implementations. In cases such as these, CJD supports comments not only for the artifact as a whole but for its subartifacts as well.

Namespaces are a kind of artifact, too, and can have their own CJD comments. Finally, a project doesn't currently rate as an artifact per se, but it can have usefully an overview comment for the project as a whole. We note in this regard that a Leiningen  project.cjd file is a dandy place to store such a thing. 

Can I extend CJD to support artifacts I've defined?

Absolutely. A first pass at instructions is slowly but surely appearing in the user guide .

Do please keep your expectations under control, however. Adding support for artifacts with no internal structure is very easy. Adding support for increasing amounts of internal structure---parameters, fields, methods, etc.---is correspondingly more work. Expect to have to build parsers for the input and HTML generators on the output side (though you may be able to leverage existing developments in either case). You may even find yourself doing strange things like adding code to support reflection on your artifacts' structure (been there, done that). None of which is impossible, of course, or even likely to strain your programming abilities, but it will feel like real work.

Can CJD use my existing docstrings in lieu of CJD comments?

Yep. It even does an arguably semi-credible job of formatting them, though the results are not nearly as pretty as CJD comments (as an example, you can see the results of running CJD on the Clojure source here). It doesn't currently do docstrings by default, though: you'll need to specify :docstrings true in the options to CJD.

I think your output format stinks. Can I invent my own?

Knock yerself out, dude. You've got several opportunities.

If your concerns are merely surface-cosmetic, like if you don't like the fonts or color scheme, you might get away with just overriding declarations in CJD's base CSS documents. Similarly, CJD provides a few hooks for such purposes as adding custom headers and footers. The user guide describes how to do this kind of customization.

But if you've got bigger aspirations...

CJD operates in about a two-and-a-half pass fashion, where the first pass involves reading and parsing input and creation of a rough abstract syntax tree for the Clojure program per se---or forest, actually, as there's a tree (shrub, more like) per namespace. This serves as the input to the HTML-generating back end. CJD comments per se are currently parsed as an immediate precursor to HTML generation.

All of which means you could either hack the existing back end (ugly, but not intractable), or write your own. For this purpose, the source code is its own documentation. Enjoy!

Does CJD work for ClojureScript or Clojure on CLR?

Nope. Currently CJD only works for Clojure/JVM. Current best guesstimate is that there are no inherent limitations in extending support to Clojure/CLR, but supporting ClojureScript involves some rather scary problems. In either case, don't hold your breath: doing these things involves, you know, like, real work.

FWIW, the principle perceived problem with ClojureScript support has to do with the fact that CJD's current implementation operates using the target program's classpath, and dynamically loads each namespace it encounters. This allows the CJD implementation to use Clojure's substantial built-in support for resolving the various names it encounters, for such purposes as determining link destinations. This would seem to pose no problem in a CLR environment, but feels kinda sorta problematic for ClojureScript.

... on theological issues

You really don't like docstrings, do you?

Docstrings are OK. Really. It's just that they're, um... a trifle limited. Moreover, if you're coming to Clojure with a Java background, you may not find them particularly loveable.

Actually, docstrings have a perfectly reasonable use case. If you're plodding along in a REPL and you suddenly find yourself asking something like, "So, what was the third argument to function xyz?", you can just type in (doc xyz), and out pops the answer. Or, to put it another way, docstrings feel like they're possibly adequate as a sort of quickie aide-mémoire.

By stretching things a bit, we might even concede that they're an adequate documentation form for small functions, such, for example, as make up much of the clojure.core namespace. But the suggestion about docstrings here (see the :doc bullet)---"a string containing short (1-3 line) documentation for the var contents"---suggests what feels like a really appropriate limitation.

By the time we get to dealing with an artifact as complicated as, say, defrecord, we're clearly in a different ballpark. That's a pretty big chunk of documentation, and there are clearly much larger artifacts out there.

Another perceived problem with docstrings is that even when the information is there, it's not always easy to find. Our poster child (whipping boy?) for this has always been the docstring for reduce: it's easy enough to conclude that its first parameter is a function. Now... what are that function's parameters, and in what order? Well, the information is there; it merely takes a bunch of digging to extract. Which, after you've had to look it up several times not only ceases to be fun, but starts costing a goodly chunk of productivity. Point is, adding some structure to the documentation and arriving at a more or less consistent presentation can make life much easier for the reader.

Aside from all of the foregoing, some of us simply detest having to write and maintain hand-formatted simple-text-only documentation. And it has to be expressed as a string, forsooth. (The author's reaction on discovering docstrings, incidentally, was, "Oh, joy. State-of-the-art embedded documentation... for 1985."*) Javadoc is arguably a step above this (state-of-the-art for 1995), as it's at least heard of things like links and multiple fonts and emphasis in text and suchlike. So, you can view CJD as an attempt to propel Clojure's internal documentation capabilities slightly less far into the past.

 

* As it turns out, this off-the-wall snark seems to be fairly accurate. The 1978 edition of the InterLisp Reference Manual doesn't obviously have anything like doc strings. Common Lisp, as witnessed by Guy Steele's 1990 Common Lisp: The Language, 2nd Ed., seems to... and gives 1988 as the year it entered the standard.  So, 1985 can't be that far off.

Should I feel evil, unclean, or insufficient if I don't like CJD?

Nope. Whatever works for you, my friend. Seriously! We all have our own preferences, our own experiential history, our own styles, our own senses of what works and what doesn't, all for our own reasons. Far be it from us to inflict anything upon you. If docstrings (or whatever else) works for you, dynamite. As with all programming-related endeavors, we see this as an engineering exercise, not a religious crusade.

CJD was developed with exactly one requirement, which was satisfy its author's urge to write copious quantities of documentation so that (1) he has a forum to convince himself that he knows what he's doing, (2) so that when he dives back into a Clojure project after months (or minutes) away from it, he has something better than a snowball's chance of recovering a clue as to what the code was all about, and (3) so he can narcissistically admire the pretty output. He's magnanimously inflicting CJD on the world at large just in case there are other unfortunates out there with similar tendencies. If you're one of these, welcome to the club! If not, it won't be held it against you. Or at least, not much.

So, why didn't you just copy Javadoc?

So, why didn't Clojure just copy Java? Duh! (Sorry, couldn't resist that.)

Javadoc is the (distant) point of departure for CJD---the inspiration, if you will. Strategically, Javadoc feels like it gets most of the big-picture things about right---like defining a reasonably structured framework for commenting, and generating hypertext-based output (this is the Internet age, after all).

Javadoc has lots of little limitations and annoyances. Like, having to type <p> to start a paragraph... but @param to start a parameter description. Or, heaven help us, {$link ..whatever..} to insert a link (is that even the right syntax? Can't ever remember...).  But that's the petty-annoyance stuff.

A few major issues: first, we use editors that are highly tuned for editing Clojure code. Stands to reason, then, that the documentation syntax would also be Clojure-ish in form, so our editors can help us out.

Second, artifacts arising in Clojure are often recursively defined. If we define a function, it has parameters and a return value, so we'd generally expect to document those. Now, what if one of those parameters is also a function? Being as there are likely very explicit expectations about that parameter-function's parameters, return value, and behavior, it surely seems like its required behaviors and parameters and return value ought to be documented, too. And that function might have a parameter that's a function, too... Well, hopefully you get the point.

Same idea applies to data structures, too, and that's a huge driver here. For example, one of Clojure's best features is the ability it bestows on a programmer to easily structure and destructure data on an ad-hoc, as-needed basis. This makes it trivially easy, for example, to return and use multiple values from a function---which capability we personally use pervasively. But effects like multiple return values merely increase the demand for documentation (like, what order are a function's return values in? Same question like for parameters!), and the need for documentation structures to support them.

Third, syntactically speaking, Java doesn't extend. It defines a fixed set of stuff---classes, interfaces, enums, and annotations---and that's the sum total of all she wrote, period, finis, requiescat in pace, end of story, amen, over and out. Clojure, however, has macros, and macros can spawn all sorts of fun new artifacts, any of which may have their own needs and aspirations when it comes to documentation. From this it follows that CJD needs to extensible as well. Like, if your program has a defwhatsit macro and you're creating whatsit thingies all over the place, then CJD needs to be able to allow you to add the wherewithal to generate pretty documentation for your whatsits as such. 

Bottom line here is that Javadoc feels like a reasonable point of departure; but having departed, CJD follows its own path.
