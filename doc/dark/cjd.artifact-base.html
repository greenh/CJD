cjd.artifact-base

Provides the basis for defining artifacts, objects that represent Clojure constructs susceptible to carrying CJD comments.

Artifact has two major interpretations. From one perspective, artifacts are representations of various defined Clojure constructs, such as vars, functions, macros, records, etc., and contain content describing the construct, among which is the CJD comment for the form. The definition of an artifact may include second-level structures in the form of subartifacts. These are used, for example, to represent methods.

From another perspective, artifacts are nodes in a rough abstract syntax tree parsed from a Clojure source file. This tree (or forest of trees, in the case of a multi-file CJD run) inform all subsequent steps of generating the finished documentation.

defartifactdefmacro in cjd.artifact-base

(defartifact [artifact‑name keying‑symbol descriptive [& extensos] [& local‑fields] parse‑fn & local‑additions])

Defines a CJD "artifact", an object class that representes a specific kind of entity declared in Clojure source code by a well-known symbol. Defining an artifact wires the artifact type into the CJD infrastructure so that will be recognized during processing, and creates the necessary structures to record salient information about instances of the artifact as they're encountered. As a consequence, defartifact represents a major extension point for CJD.

clojure.core specifies a large selection of entities that give rise to artifacts, such as variables (declared by def), functions (defn), macros (defmacro), and many others. These are mostly specified in cjd.core-artifacts.

Invoking defartifact to define an artifact gives rise to several effects.

  • It defines a Clojure record type (using defrecord) which has the name given by artifact-name. The current implementation is based on the constructo and extenso mechanisms to generate the record.

  • It generates a method and adds it to the cjd-artifact multimethod. This method handles the boilerplate processing for the artifact, and is called by the CJD system whenever it encounteres a top-level form headed by keying-symbol while processing source files. Additionally, the method calls parse-fn to do any further parsing of the source form.

Arguments

artifact‑name — A symbol denoting the name of the artifact.

keying‑symbol — The symbol used to identify instances of the artifact, e.g. def, defn, defmacro, defartifact, and the like.

descriptive — A very short string that describes what this artifact represents, e.g. "function" for defn, "macro" for defmacro, etc.

extensos — A (possibly empty) sequence of extensos to be included in the artifact, optionally including field specifications and initializations. This uses the same specificational conventions as defconstructo.

local‑fields — A (possibly empty) sequence of local fields, possibly including initialization values. This follows the same specificational conventions as defconstructo.

parse‑fn — A function of the form (fn [form]), which the CJD infrastructure invokes whenever it encounters a top-level form with the type declared by the artifact.

form — A form with the functor given by keying-type.

Returns — An object of the type given by artifact-name.

local‑additions — Local method implementations, followed by local implementations of protocols, interfaces, or java.lang.Object methods. This follows the same conventions as in defconstructo.

Returns

The class object generated for the artifact, ala defrecord.

See also

The cjd.core-artifacts namespace contains many examples of the use of defartifact.

defsubartifactdefmacro in cjd.artifact-base

(defsubartifact [subartifact‑name [& extensos] [& local‑fields] parse‑fn & local‑adds])

Defines a CJD subartifact, an object class that defines the representation of some type of documentable element subordinate to an artifact.

Artifacts are used to represent top-level forms such as defined functions, macros, protocols, or many others. Subartifacts, on the other hand, are used to represent subordinate elements of artifacts, such methods within a protocol. As such, a subartifact always has a parent object, which may be an an instance of artifact or componnt, as appropriate.

Arguments

subartifact‑name — The name of the subartifact type being defined.

extensos — A set of extensos used to construct the subartifact.

local‑fields — A set of fields to be defined as part of the subartifact. These fields are in addition to any defined by the extensos, and may include initializations, as defined for constructos.

parse‑fn — A function of the form (fn [parent source‑form & params]), where

parent — The parent artifact (or subartifact).

source‑form — A form from the source input stream to be parsed. A function that the parse-subartifact-name calls to parse source forms.

params — Additional parameters as needed; these are by agreement between the parent artifact's parsing function and the subartifact's.

Returns — An object of the subartifact's type.

local‑adds — A (possibly empty) set of methods to be defined for the the subartifact.

These local methods may be followed by a sequence of protocol interface names, or java.lang.Object, each followed by method implementations. Note that this is exactly the same syntax used for record and type definitions!

Returns

The class object for the record defined by the subartifact.

simple-artifact-parserdefn in cjd.artifact-base

(simple‑artifact‑parser [_])

Minimal parsing function for use with defartifact.

In situations where an artifact type has no documented internal structure (or when you're going to implement it all "later", but would like the documentation for the top level element now), simple-artifact-parser is a handy way to get artifact objects appropriately instantiated without further ado.

Note that this only works if the artifact is declared with no extensos or local fields.

Returns

An appropriate artifact. The specific artifact type is determined from the context of use by defartifact.
